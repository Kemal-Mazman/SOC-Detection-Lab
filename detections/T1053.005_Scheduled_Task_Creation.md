Scheduled Task Creation Detection
MITRE ATT&CK Mapping
FieldValueTacticPersistence (TA0003), Execution (TA0002), Privilege Escalation (TA0004)TechniqueScheduled Task/Job: Scheduled Task (T1053.005)Data SourceSysmon Event ID 1 (Process Creation)SeverityMediumPlatformWindows
Description
Detects the creation of new scheduled tasks via schtasks.exe /create. Adversaries abuse the Windows Task Scheduler to execute malicious payloads at system startup, on a recurring schedule, or in response to specific events. This is one of the most common persistence mechanisms observed in real-world intrusions, used by threat actors including LockBit, BlackCat/ALPHV, and APT29 to maintain access after initial compromise.
Scheduled tasks survive system reboots, can run under different user contexts (including SYSTEM), and blend in with legitimate Windows administration — making them a high-value persistence mechanism for attackers.
Detection Logic (Splunk SPL)
splindex=main sourcetype="XmlWinEventLog:Microsoft-Windows-Sysmon/Operational" "schtasks" "/create"
| rex field=_raw "CommandLine'>(?<CommandLine>[^<]+)"
| rex field=_raw "User'>(?<User>[^<]+)"
| rex field=_raw "ParentImage'>(?<ParentImage>[^<]+)"
| rex field=_raw "Image'>(?<Image>[^<]+)"
| where NOT match(CommandLine, "(?i)(\\\\Microsoft\\\\|Windows\\\\Task|SCCM|ConfigMgr)")
| table _time, User, Image, CommandLine, ParentImage
| sort - _time
How the Attack Works

Attacker gains initial access to a Windows endpoint (e.g., phishing, exploit)
Attacker creates a scheduled task to execute their payload on a recurring schedule:

   schtasks /create /tn "UpdaterTask" /tr "cmd.exe" /sc minute /mo 1

The task persists across reboots and executes the payload (reverse shell, C2 beacon, ransomware dropper) at the defined interval
Attacker maintains persistent access without requiring re-exploitation

Tuning & False Positives
Expected False Positives

System administrators creating legitimate maintenance tasks
Software installers registering update tasks (e.g., Google Update, Adobe)
SCCM/ConfigMgr deploying scheduled tasks for patching
Group Policy deploying scheduled tasks across domain

Tuning Recommendations

Baseline normal scheduled task creation in your environment during the first 7 days
Exclude known legitimate task names and paths via the where NOT match filter
Investigate tasks with suspicious /tr values: PowerShell, cmd.exe, encoded commands, temp directories, or external URLs
Alert on tasks created by non-administrative users
Correlate with other persistence indicators (registry run keys, services)

Analyst Triage Guide
When this alert fires, investigate:

Who created the task? — Is this user expected to create scheduled tasks?
What does the task run? — Check the /tr parameter. Cmd.exe, PowerShell, or binaries in Temp/AppData are suspicious
What is the schedule? — Tasks running every minute or at logon are common for C2 beaconing
What is the parent process? — Was schtasks spawned from cmd.exe, PowerShell, or an unusual process?
Is there related activity? — Check for other persistence mechanisms or lateral movement around the same timeframe
